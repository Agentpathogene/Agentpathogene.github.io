HORIZONTALL_WRITEUP

OS: Linux
> 01_ENUMERATION

On commance la reconnaissance : avec un scan nmap -p- et -A ; seuls les ports 22 (SSH: secure shell protocole) et 80 (HTTP : hyper texte transfert protocole) sont ouverts on a donc une application web qui tourne sur la machine. Par ailleurs le scan nmap révèle un nom de domaine : horizontall.htb, on configure la resolution DNS dans /etc/hosts.

On analyse soigneusement l application web :
on arrive a 2 conclusions :
1_ les entrées utilisateurs sont sécurisées
2_ le fuzzing ne révèle rien de compromettant.
On analyse fichiers js : ce qui révèle un sous domaine , "api-prod" ; une fois la redirection configurée dans /etc/hosts , on arrive sur une application qui utilise le CMS strapi : CMS qui est vulnerable a une RCE sans authentification sur la page de connexion ! .La page de connexion elle est visible sur /ADMIN un fuzzing simple avec dirb gobuster ou encore ffuf permet de la découvrir!!!

> 02_EXPLOITATION

Exploitation de la RCE :
On utiliste l'exploit de exploit DB afin d'obtenir un shell sur la machine victim , une fois l'accès obtenu , on verifie quels utilisateurs ont un acces shell avec : cat /etc/passwd | grep sh .
On apprend que l'utilisateur devloper en plus de root et de notre utilisateur courant ont un acces shell ; et , sur le repertoir /home/devloper on à accès au flag user de la machine ....!!!

> 03_ESCALADE_DES_PRIVILEGES

On a pas de binnaire executable en sudo ni de credentials.
On regarde ce qui tourne en local avec: ss -tulpn , mais on ne parviens pas a lister a cause de notre shell restrictif
Pour passer a un shell ssh stable et non restrictif :
On va alors cree un dossier .ssh dans le repertoire principal de notre utilisateur courant sur la machine victim , et ensuite y mettre notre clé publique rsa dans un fichier authorized-keys pour ensuite pouvoir se connecter en ssh avec la machine cible avec un shell qui n est pas restreint ce qui nous permettra d' énumerer ce qui tourne en interne

Sur le port 8000 : On decouvre , (grace a une requete curl) une application qui utilise Laravel et une vielle version de php.

Ensuite on configure un tunneling ssh simple pour avoir accès a l'application vulnérable depuis sa machine attaquante...

Une fois la redirection effectuée :
on se renseigne sur la version de Laravel on apprend qu'elle est vulnérable a une RCE (CVE-2021-3129) on peut donc executer des commandes systems en tant que root. ( Car le service tourne en root , c est du moins ce que j'en ai deduis car qu en j'ai voulu savoir quel utilisateur la faisait tournee j'ai eu une erreur de permission ...)
Pour l'exploiter de nombreuses pocs son disponibles sur github , pour cette machine la meilleur est https://github.com/nth347/CVE-2021-3129_exploit ! enjoy the root !
